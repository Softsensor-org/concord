# Governance Policy

This is the canonical execution policy for a project using this coordination scaffold.

Version: v0.5.0 (2026-05-16)

Layer label: `current-enforced-policy`

Use this file for:
- enforced governance rules
- canonical file ownership
- authority order and lifecycle constraints

If another governance doc conflicts with this file on an active rule, this file wins.

## 1) Goals

This governance is optimized for:
- safe parallel work across governed repos and `coord`
- one canonical board
- deterministic ticket state
- low-friction startup for new agents

## 3) Authority Order

When instructions conflict, use this order:
1. Human-admin
2. `coord/GOVERNANCE.md`
3. Ticket prompt in `coord/prompts/`
4. Repo-local `AGENTS.md` files
5. Tool-specific shim files (`CLAUDE.md`, `CODEX.md`, `GEMINI.md`)
6. Orchestrator cycle decisions
7. Agent judgment

Rules:
- Orchestrator decisions cannot override human-admin instructions, governance, or ticket prompts.
- If an orchestrator instruction conflicts with a higher authority, stop and log the conflict in `coord/QUESTIONS.md`.

## 3.1) Agent Entry Paths

This governance is agent-agnostic. Use `coord/AGENT_PATHS.md` as the single practical guide for Codex, Claude, and Gemini native entry flows.
All governed mutations still converge through the shared CLI or MCP core regardless of which frontend or wrapper path an agent uses.

Rules:
- the preferred operator surface is the small facade defined in `coord/VERB_CONTRACT.md` and implemented at `coord/scripts/agent`
- this policy references `coord/VERB_CONTRACT.md` for the canonical facade verb list instead of restating that list here
- `coord/scripts/agent` is a thin adapter over the existing governance core; it must not duplicate lifecycle logic from `coord/scripts/gov` or the MCP layer
- raw `coord/scripts/gov` remains the admin, CI, break-glass, and deep-inspection surface
- Choose one native path per session.
- Cross-agent handoffs are allowed only through clean `claim` / `resume` semantics.
- Tool-specific wrappers do not override governance, ticket prompts, or board state.
- comparative-strength routing across Claude, Codex, and Gemini should use `coord/AGENT_CAPABILITY_REGISTRY.md`, not prompt folklore

## 3.2) Agent Boot, Retrieval, and Externalization Contract

This section defines the cold-start contract for every governed agent. It adds no
new subsystem: agents resume from existing governed artifacts and write durable
learning back into those same artifacts.

Chat memory is non-authoritative. A transcript, model memory, or remembered
prior answer may help locate relevant material, but it never outranks the board,
governance journal, plan records, prompts, repo gates, feature proofs, accepted
ADRs, or product requirement sources.

Boot sequence:
1. Read the thin entry shim: root `AGENTS.md`, then the active tool shim
   (`CODEX.md`, `CLAUDE.md`, or `GEMINI.md`) when present.
2. Resolve canonical precedence from this file, then the ticket prompt,
   repo-local `AGENTS.md` files, tool shim, orchestrator cycle decision, and
   agent judgment in the authority order above.
3. Bind identity before mutation. Use `coord/scripts/gov agentid --assign` for a
   fresh session, then `start`, `claim`, or `resume` the ticket through
   governance as appropriate for the current state.
4. Retrieve the minimum governed context before planning.
5. Plan, execute, verify, and move through the review gate.
6. Externalize durable learning before closeout so the next cold-start agent can
   resume without relying on chat memory.

Minimum context for nontrivial tickets:
- `coord/scripts/gov explain <ticket>` as the first ticket read;
- the ticket prompt or waiver and current board row;
- the active plan record, prework/context-pack, and ticket-local notes when they
  exist;
- relevant recall or memory-claim proposals, treated as leads until verified
  against governed artifacts;
- accepted ADR references, ADR proposals linked from the ticket, and product
  requirement sources that the ticket cites or modifies;
- business-discovery packs or requirements packs for discovery/product-shaping
  work;
- repo-local `AGENTS.md` files and gate docs for every repo touched by the
  declared write scope.

Externalization requirements:
- record plan updates, requirement-closure evidence, review cycles, feature
  proofs, and repo-gate results through governance where the CLI supports them;
- link accepted ADRs or record an ADR proposal when the work changes durable
  architecture, coordination semantics, or operator policy;
- record memory-claim proposals only for reusable learning that has been
  verified against current governed artifacts;
- resolve questions, decisions, and reflections in the appropriate governed
  logs instead of leaving them only in chat;
- close out with enough evidence that a new agent tomorrow can run
  `coord/scripts/gov explain <ticket>`, read the linked artifacts, and understand
  what was done, what remains deferred, and which gates proved it.

## 4) Ticket Status Model

### Legal statuses

- `todo`
- `doing`
- `doing (blocked: <reason>)`
- `review`
- `done`
- `deferred`
- `superseded`

### Legal transitions

```text
todo -> doing
doing -> review
doing -> doing (blocked: <reason>)
doing (blocked: <reason>) -> doing
review -> done
review -> doing
todo -> deferred
deferred -> todo
any -> superseded
```

Guarded wrong-start exception:
- `doing -> todo` is allowed only through `gov unstart <ticket-id>`
- `gov unstart` is same-owner only and must fail closed once the ticket has accrued review, landing, plan, or workspace evidence that should remain auditable on the board
- foreign stale-state cleanup remains an explicit admin path (`lock-abandon`, `takeover`, or orchestrator cleanup with human-admin override)

## 5) Ownership and Locks

- Only one active owner per `doing` ticket.
- Code-edit tickets require a lock at `coord/.runtime/locks/<ticket>.lock`.
- Lock files must include (v2 owner-authoritative model — see §10.3):
  - `owner`
  - `ticket`
  - `status`
  - `repo` (configured repo path from `coord/project.config.js`, or `coord`)
  - `branch`
  - `worktree`
  - `started_at_utc`
  - `heartbeat_utc`
  - `head`
  - `identity_model` (`"v2"` on locks written under the durable identity
    channel; authority is the **owner lease**, not a session token)
- **Legacy fields** `session_id` / `thread_id` are no longer authority.
  They remain readable on pre-v2 locks for ≥1 release (ignored for
  authority, `gov doctor`-warned) and MUST NOT be written as authority on
  new v2 locks. GCV-1 removes no code; deletion of the legacy
  session/thread machinery is a later, fleet-migration-gated release.
- this section defines the high-level required lock fields only; `coord/docs/AGENT_ID_COLLISION_RECOVERY_POLICY.md` Section 3 is the normative source for the full lock schema semantics, including complete-lock validation, `head` meaning, and Repo X sentinel carve-outs
- for coord-owned tickets, `head` must use the sentinel value `coord-no-git-head`
- for git-backed repos, `head` is the governed worktree commit anchor and should match the governed worktree `HEAD` when the worktree is on its canonical branch
- `doing` tickets must have a valid lock.
- Non-`doing` tickets must not have a lock.

## 6) Repo Codes

Repo codes are defined in `coord/project.config.js`. Each code is a single uppercase letter mapping to a repo directory. `X` is always reserved for coord/cross-repo work.

Default project config:
- `B` = first repo (default: `backend`)
- `F` = second repo (default: `frontend`)
- `X` = cross-repo, design, planning, or coord-owned work

Additional codes can be added for projects with more repos (e.g., `M` = mobile, `W` = worker). The governance CLI, board validator, and MCP server read project config through `coord/paths.js`; edit `coord/project.config.js`, not engine-managed `paths.js`.

## 7) Worktree Rules

- Fleet mode is worktree-per-agent by default. Every governed agent must work from a repo-local or coord-owned worktree under `.worktrees/<agent>/<ticket>/`; a shared checkout/runtime is not a supported multi-agent execution surface.
- `gov start` provisions the ticket worktree automatically. Repo-backed tickets use the product repo's `.worktrees/<agent>/<ticket>/`; repo `X` tickets use a real coord git worktree under `coord/.worktrees/<agent>/<ticket>/` with a worktree-local, gitignored `coord/.runtime/` scaffold.
- A second heartbeat-fresh governed session in the same checkout/runtime is refused by default. Use a separate git worktree with its own `coord/.runtime`; pass `--allow-shared-worktree` only for deliberate single-agent/local or orchestrator-controlled exceptions where the operator accepts the shared-runtime risk.
- Repo `X` work is split by declared file surface:
  - safe coord code/doc surfaces such as `coord/scripts/`, `coord/docs/`, `coord/product/`, `coord/ui/`, `coord/GOVERNANCE.md`, and repo-local agent guide files may be scheduled in parallel waves when declared files are disjoint;
  - global coordination state such as `coord/board/`, `coord/.runtime/`, `coord/.worktrees/`, `coord/prompts/`, `coord/rendered/`, `coord/PLAN.md`, and `coord/QUESTIONS.md` remains single-writer and must be scheduled alone.
- Repo `X` tickets without declared files are treated as potentially global-state-affecting and scheduled alone.
- The repo root should stay clean.
- Never overwrite another agent's work.
- If unrelated dirty files are found, stop and log the conflict in `coord/QUESTIONS.md`.

## 7.1) Branch Promotion Model

For governed repo work (any repo code except `X`):
- ticket branches land on `dev`
- `dev` is the canonical integration branch used for governed landing evidence, ancestry audit, and feature-proof checks
- `main` is a promoted branch only; do not advance `main` as part of normal ticket closeout
- update `main` only through an explicit promotion step from `dev`, separate from ticket `review -> done`

Rules:
- no-PR and PR-backed ticket closeout must record landing evidence against `dev` unless a human-admin explicitly overrides the base branch
- agents must not fast-forward, merge, or push `main` during ordinary ticket landing just to keep it aligned with `dev`
- if `main` needs to move, treat that as a promotion action after the relevant `dev` baseline has already landed and stabilized

## 7.2) Git Management

For governed repo work (any repo code except `X`):
- `gov start` is the earliest git safety gate; if `origin` is configured, refresh `origin/<base>` before creating a new governed branch
- `--offline` is the explicit escape hatch for start-time branch creation when network refresh is unavailable or intentionally skipped; in offline mode, governance may rely on locally cached refs
- local preflight must fail closed on detached HEAD and on direct work from `dev` or `main`
- commit subjects must already contain the ticket ID; governance validates this but does not rewrite the subject automatically
- targeted staging is the default expectation for governed commits; broad staging is only allowed through an explicit override (`--all`)

Merge-method policy:
- default to `squash` for ordinary ticket landing
- use `rebase` only when the repo or hosting policy requires linear history and the ticket branch is intentionally reviewable as a small linear series
- use `merge` only by explicit exception when preserving branch topology or multi-commit ancestry is itself part of the reviewed change

## 8) Start Gate

Before moving a ticket to `doing`:
1. Read `coord/AGENT_STARTUP_CHECKLIST.md`.
2. Confirm the ticket exists in `coord/board/tasks.json`.
3. Confirm the status transition is legal.
4. Confirm there is a prompt mapping in `prompt_index` or an explicit waiver in `waiver_index`.
5. Verify the ticket prompt's declared preconditions against the target repo's integration branch (see Prompt Preconditions below).
6. Bootstrap or materialize canonical plan state.
7. Auto-seed startup / traceability attestation at start time.
8. If the ticket type is `test`, `contract`, or `infra`, record baseline reproduction before the start can complete.
9. Acquire the lock and create the governed workspace.
10. For repo-backed tickets, refresh `origin/<base>` before creating a new branch when `origin` is configured; use `--offline` only as an explicit local-ref fallback, then print reproducible dependency bootstrap guidance.
11. Run `node coord/board/board.js sync`.

### Prompt Preconditions

A ticket prompt may declare an OPTIONAL `## Preconditions` section listing concrete artifacts the ticket claims **already exist** and will modify or gate — for example a security ticket gating an existing route, or a refactor ticket changing an existing symbol. The start gate parses that section and verifies each declared artifact against the target repo's integration branch (`origin/<base>`, falling back to a local `<base>` branch) using cheap git lookups only — no build, no checkout:

- a file path is verified with `git cat-file -e <ref>:<path>`
- a route or text literal is verified with `git grep` against the integration ref
- a `symbol:<path>#<literal>` entry requires both the file and the literal

If any declared precondition does not resolve, the start gate fails **before** any lock or worktree is created, with a clear stale-prompt error naming the unresolved artifact(s). This catches prompts authored ahead of a dependency that has not landed.

Rules:
- The section is optional. A prompt with **no** `## Preconditions` section declares nothing, verifies nothing, and starts normally — existing prompts are unaffected.
- Each precondition is a path / route / symbol. Use the explicit `path:` / `route:` / `text:` / `symbol:<file>#<literal>` prefix form, or a bare token (a leading `/` is treated as a route literal; a path-shaped token is treated as a file path).
- `## Likely Files` are NOT preconditions and never block a start — those are frequently files the ticket will create.
- Declare an artifact as a precondition only when the ticket premise is false without it.

### Prompt Waivers

When a ticket does not have a prompt mapping in `prompt_index`, it must have an explicit waiver in `waiver_index` before it can move to `doing`. Waivers are recorded in `coord/board/tasks.json` and must include:
- `code`: the waiver type (currently `prompt_coverage`)
- `reason`: why a prompt is not required
- `recorded_at`: when the waiver was recorded
- `recorded_by`: who recorded it

Prompt creation and registration:
- low-ceremony `gov file-ticket` remains valid without prompt coverage
- when a ticket should be start-ready at creation time, use
  `coord/scripts/gov file-ticket ... --with-prompt` to create
  `coord/prompts/tickets/<ticket-id>.md` and register `prompt_index` coverage
  in the same governed mutation
- when an existing ticket needs its missing canonical prompt, use
  `coord/scripts/gov register-prompt <ticket-id> --create`
- do not hand-create prompt files and then repair sealed-state drift unless
  explicitly recovering from an interrupted command; the governed create/register
  path is the normal atomic path
- a ticket already registered to a different prompt path must fail closed unless
  the operator passes an explicit replacement/force flag

## 9) Review Gate

Before moving `doing -> review`:
- run the relevant repo checks defined by `coord/product/TESTING_AND_GATES.md` and `coord/product/LOCAL_AUTOMATION_AND_GATES.md`
- record verification commands in the plan record / plan ledger
- record explicit requirement-closure evidence in the plan record:
  - `Ticket ask: ...`
  - `Implemented: ...`
  - `Not implemented: ...`
  - `Deferred to: <ticket-id or none>`
  - `Closeout verdict: complete`
- for governed code-repo tickets at or after the feature-proof cutoff, record canonical `Feature proof` entries in the plan record:
  - `path:<repo-relative-file>`
  - `symbol:<repo-relative-file>#<literal-or-symbol>`
  - `text:<literal>`
  - `route:<literal>`
- perform at least 4 self-review cycles for substantive code-repo changes, covering:
  - contract/state invariants
  - auth/security/failure modes
  - tests/operability/performance
  - requirement closure
- `coord`-only governance changes still require at least 3 structured self-review cycles
- if a governed code-repo ticket was merged to the canonical base branch before `doing -> review`, treat that as a sequence violation:
  - do not use `supersede`
  - do not hand-edit board state
  - repair it with `coord/scripts/gov finalize <ticket-id> --no-pr --already-landed --landed "<canonical-branch closeout proof>"`
- release the lock in the same cycle
- sync the board

## 10) Done Gate

Before moving `review -> done`:
- confirm merge / closeout evidence
- record the canonical landed commit, and record the source commit too when it is still available
- when a ticket is satisfied by later work, record that explicitly via `fulfilled_by_ticket` and/or `fulfilled_by_commit_sha` instead of relying on narrative notes
- confirm the landed commit is recorded and is an ancestor of the canonical base branch (`dev` unless explicitly overridden)
- run any required canonical feature-proof audit for governed code-repo tickets
- remove the governed worktree for code-repo tickets
- keep any follow-up work in a new ticket; do not reopen `done` tickets
- when a repo has a configured remote, prefer PR-backed landing; use no-PR closeout only as an explicit exception
- do not advance `main` during ordinary ticket closeout; promote `dev -> main` only as an explicit post-landing action
- `supersede` is only for obsolete work that is not already landed on the canonical base branch; landed work must close through the governed review/done path
- sync the board

## 10.1) PR Merge Conflict Resolution

If a PR cannot merge due to conflicts:
- the owning agent must rebase the ticket branch against the current base ref (`dev`)
- verify no new test failures are introduced by the rebase
- force-push the rebased branch with `git push --force-with-lease`
- confirm the PR merge state returns to `MERGEABLE` before retrying `gov land`
- do not use the `--admin` flag to force-merge over conflicts

If a base ref has diverged significantly or the branch is stale:
- consider opening a new PR from a fresh branch
- supersede the conflicted ticket only if the underlying work is obsolete

## 10.2) Revert and Hotfix Policy

- reverting landed work requires a new governed ticket; do not reopen or rewrite the historical ticket that originally landed
- ordinary urgent fixes still land on `dev` first, then promote through the normal `dev -> main` flow
- direct `main` hotfixes are human-admin only and must record the override reason explicitly
- every direct `main` hotfix must open a follow-back ticket that replays or verifies the fix on `dev` before the next promotion
- a direct `main` hotfix does not count as restoring normal branch alignment until the follow-back work is landed on `dev`

## 10.3) Identity and the Owner Lease (v2)

> **Rewritten for GCV-1 (identity v2).** This supersedes the prior
> tty-fingerprint / "one live conversation per board" / explicit-
> `CLAUDE_SESSION_ID` rule. That rule existed *because* "a provider [had
> not yet exported] a stable per-conversation child-process identity."
> That premise no longer holds: Claude Code's `SessionStart` →
> `CLAUDE_ENV_FILE` channel is the durable, explicit per-conversation
> primitive that was missing. The reversal is justified by the primitive
> becoming available — exclusivity is **strengthened** (an owner lease),
> not relaxed.

**Core invariant.** A ticket may be mutated only by its recorded `owner`,
through exactly **one fresh live instance** holding that owner's lease.

- **Durable identity** comes from the provider session
  (`COORD_PROVIDER_SESSION_ID`, from the Claude `SessionStart` hook via
  `CLAUDE_ENV_FILE`). **Live exclusivity** is a per-attach
  `COORD_INSTANCE_ID`. `provider_session_id`, `transcript_path`, pid/host
  are **evidence/diagnostics only — never authority**.
- **The mutex is the owner lease**, not a session token: at most one live
  registered instance per `owner`, regardless of how many provider
  sessions exist.
- **Liveness** is heartbeat + TTL (default 15m, `COORD_INSTANCE_TTL`).
  Stale instances are reapable and never block. **Owner-level
  split-brain** (≥2 fresh instances for one owner) fails closed.
- **Same-owner fast handoff** (crash / fast restart): `gov claim --owner
  <X> --handoff [--reason <text>]` atomically journals an
  `owner-lease-handoff`, transfers the lease, and revokes the prior
  instance (which then fails closed explicitly on its next mutation — no
  silent steal). Same-owner only; cross-owner takeover is still
  `--human-admin-override "<reason>"`. Bare `--force` is a deprecated
  legacy alias only.
- **Missing/ambiguous durable channel ⇒ fail closed** with an actionable
  message (explicit-claim-only). Authority is never inferred. MCP mutating
  tools are read-only under v2 (the long-lived server cannot prove
  per-call identity).
- `.runtime/agent_sessions.json` and the v2 instance registry are runtime
  state, not canonical ticket ownership by themselves; when runtime state
  conflicts with a valid governed ticket lock, the ticket lock (owner)
  wins.
- **Migration:** pre-v2 locks with `session_id`/`thread_id` remain
  readable for ≥1 release (ignored for authority, `gov doctor`-warned).
  GCV-1 deletes nothing; removing the legacy session/thread machinery is
  a later release gated on downstream fleet migration + GCV-4.

### Identity

Use these sources together:

- runtime quick rules for agents and shims: `coord/docs/IDENTITY_RUNTIME_EXTRACT.md`
- canonical collision/recovery companion: `coord/docs/AGENT_ID_COLLISION_RECOVERY_POLICY.md`
- provider `thread_id` declarations and fallback behavior: `coord/docs/provider-thread-id-sources.md`

Rules:
- `coord/docs/IDENTITY_RUNTIME_EXTRACT.md` is the startup-sized summary for current session, resume, collision, and recovery behavior
- `coord/docs/AGENT_ID_COLLISION_RECOVERY_POLICY.md` is the full target-state policy and implementation matrix
- `coord/docs/provider-thread-id-sources.md` is the canonical declaration of provider-native `thread_id` sources and fallback stability limits
- when the current thread must abandon a collided handle without touching foreign ticket state, use `coord/scripts/gov agent-rebind --fresh`
- when human-admin explicitly authorizes rebinding a foreign doing/review ticket into the current claimed session, use `coord/scripts/gov claim <ticket-id> --transfer-to <current-agent-handle> --human-admin-override "<reason>"`
- when human-admin explicitly authorizes stale foreign-lock cleanup that returns a doing ticket to `todo`, use `coord/scripts/gov lock-abandon <ticket-id> --human-admin-override "<reason>"`
- if these files disagree, `coord/GOVERNANCE.md` defines the canonical policy surface, the runtime extract defines current quick-reference behavior, and the companion doc defines the fuller target-state model that must not be overstated as already enforced

## 10.4) Plan Record Loss Recovery

If landing evidence fails to persist after a PR is merged:
- retry the same `gov land` command; the CLI detects the PR is already merged and skips the merge step
- if retry fails, use `gov mark-done <ticket-id> --landed "<commit sha and evidence>"` to manually record the landing
- if the plan record is corrupted, use `gov recover <ticket-id>` to rebuild state from the governance event log
- if a non-live ticket regresses after `review` or `done` and no canonical `doing` lock remains, use `gov recover <ticket-id> --snapshot-restore` to replay the strongest ticket-scoped restore payload recorded in the governance journal; fail closed if the historical evidence is incomplete
- do not supersede a ticket whose PR is already merged; use the governed review/done path

## 10.5) Bounded Repair Eligibility

A ticket qualifies for the bounded repair review profile when:
- the diff is narrow and file-scoped (recorded file bounds in the plan record)
- requirement closure entries are present with explicit Ticket ask, Implemented, and Closeout verdict
- the plan record matches the current governance schema
- the ticket is not a new feature or major refactor

## 10.6) Testing Infrastructure Identification

The governance CLI identifies testing-infrastructure tickets by:
- ticket ID matching `QGATE-*` prefix
- ticket description matching testing/gate infrastructure keywords
- intended files matching testing infrastructure paths (`tools/gates/`, `tests/arch/`, `vitest.config.*`, etc.)

Testing-infrastructure tickets have different validation rules:
- feature-proof may be waived when the ticket only modifies test harness or gate runner code
- the testing-infrastructure landing audit verifies that gate scripts declared in the plan exist at branch tip

## 10.7) Pre-Landing Checks

Before running `gov land`:
- verify the PR is in `MERGEABLE` state (no conflicts, passing status checks)
- verify `gov doctor` passes for the ticket
- verify the ticket's plan record has all required fields (invariants, closure, review cycles, feature proofs)
- if the PR has conflicts, rebase per Section 10.1 before landing

## 10.8) Governance Drift Classification

`detectGovernanceProvenanceDrift` (in `coord/scripts/journal.js`) compares the governed filesystem snapshot to the last journaled snapshot. The resulting drift paths are split by `splitGovernanceProvenanceDrift` (`coord/scripts/lifecycle-board-commands.js`) into two buckets:

- `warnings` — paths under `coord/.runtime/agent_sessions.json` or `coord/.runtime/session-threads/*`. These churn from session normalization; they do not block and are not logged as a QUESTIONS.md drift-note row.
- `blocking` — every other governed file, **including** `coord/QUESTIONS.md`, `coord/PLAN.md`, and the plan records under `coord/.runtime/plans/*.json`. Pre-existing blocking drift detected at the start of a governed mutation is logged as a drift-note row in `coord/QUESTIONS.md` and surfaced by `gov doctor`.

The mutation records the pre-existing drift in the governance event journal under `details.preexisting_drift` (the drift path list) and `details.preexisting_drift_logged_to_questions` (whether the drift-note row was written), so observability is preserved. To rotate `coord/QUESTIONS.md` before applying this classification to historical rows, follow `coord/docs/QUESTIONS_ROTATION_PROTOCOL.md`.

## 10.9) Live-MCP Lifecycle Enforcement

A ticket that performs a live/production MCP operation declares it EXPLICITLY by recording a structured `live_mcp` object on its plan record:

```bash
coord/scripts/gov update-plan <ticket> --live-mcp '{"adapter":"<adapter>","operation":"<op>","operation_class":"read_sensitive","environment":"prod","scope":"<explicit scope>","redaction":"masked","approval":"human-admin","receipt_path":"coord/evidence/live-mcp/<receipt>.json"}'
```

The presence of `live_mcp` (and ONLY its presence — never a description keyword match) turns on the live-MCP lifecycle enforcement gate in `coord/scripts/live-mcp-lifecycle.js`. For a declared live-mcp ticket, move-review / closeout readiness BLOCKS until the required evidence is present:

- operation class, adapter, operation, environment, and explicit scope;
- redaction evidence when the operation class requires it (`read_sensitive`, `write_prod`, `destructive`);
- approval evidence when the operation class requires it (any class above `read_safe`);
- cleanup completion when the class policy requires it, or when `cleanup_required=true` is declared;
- a recorded receipt — either embedded (`live_mcp.receipt`, validated with the COORD-152 receipt verifier) or referenced by path (`live_mcp.receipt_path`);
- fixture/test/spec promotion evidence (`live_mcp.promotion`) when `product_impact=true`.

The operation-class policy (which classes require approval/redaction/cleanup) is sourced from `OPERATION_CLASSES` in `coord/scripts/runtime-evidence.js`; this module does not re-derive it. Governance core governs intent and evidence only — it never calls a production tool. The gate is DEFAULT-OFF: a ticket with no `live_mcp` declaration carries no new requirements and is completely unaffected. `gov explain` surfaces the same missing-evidence list under `live_mcp_lifecycle`.

## 11) Orchestrator Contract

The orchestrator is a governed operator role, not an implied convention.

Responsibilities each cycle:
- assign or recommend next tickets
- enforce max one active `doing` ticket per owner unless the board declares an explicit closeout-blocker exception
- detect stale, malformed, missing, or future-dated locks
- triage unresolved items addressed to `orchestrator` in `coord/QUESTIONS.md`
- detect blocked tickets that exceeded the SLA
- keep canonical board state, governed locks, and passive alert queues coherent before making new assignment decisions

Required command:

```bash
coord/scripts/gov orch
```

Cycle rules:
- run `coord/scripts/gov orch` before and after any meaningful orchestrator assignment or reconciliation batch
- `coord/scripts/gov orch --fix` may apply deterministic local repairs, but must leave foreign ticket cleanup report-only
- batch foreign cleanup is an explicit admin path only: `coord/scripts/gov orch --cleanup-foreign --human-admin-override "<reason>"`
- do not assign additional work while `coord/scripts/gov orch` is failing hard checks

Hard checks enforced by the orchestrator cycle:
- board validation passes
- governance doctor passes
- each `doing` ticket has a valid governed lock with a canonical worktree path
- no lock heartbeat is more than 5 minutes in the future
- no active `doing` lock heartbeat is stale by more than 24 hours
- no owner holds more than one `doing` ticket unless an explicit board exception allows it
- total concurrent `doing` tickets across all agents must not exceed 5 (WIP limit) unless a human-admin explicitly overrides; this ensures output remains reviewable
- canonical code-repo roots are not left dirty outside governed worktrees

Passive alerts that must be triaged in the same cycle:
- unresolved `coord/QUESTIONS.md` rows where `To = orchestrator` and `Resolved != yes`
- `doing (blocked: ...)` tickets older than 90 minutes based on lock `started_at_utc`
- repeated governance-drift notes that have not been reconciled into a resolved question or a governed repair action
- stale or unknown foreign worktree residue; report it by default and use explicit cleanup mode only when a human-admin authorizes batch cleanup

Cycle close criteria:
- either the hard checks pass, or the orchestrator records why a human-admin waiver is required
- unresolved passive alerts are either answered, reassigned, split into follow-up tickets, or explicitly left as acknowledged debt in `coord/QUESTIONS.md`
- if stale or conflicting ownership is found, resolve it in the same cycle with `resume`, `recover`, `reconcile`, reassignment, or legal status repair

## 11.1) Safety Properties

This section documents the enforcement mechanisms that underpin the governance model.

### Lock Atomicity

Governance locks use POSIX `mkdir` for acquisition, which is atomic at the OS level. Two agents cannot simultaneously acquire the same lock. If a process crashes after acquiring a lock but before releasing it, the lock is reclaimed automatically after staleness thresholds are exceeded (24 hours for ticket locks, 2 minutes for runtime locks).

### Journal Integrity

The governance event journal (`coord/.runtime/governance-events.ndjson`) is append-only. The system uses it to detect board regression (e.g., a bad merge rolling back ticket statuses). Behavior on journal state:
- **Missing journal**: treated as uninitialized; auto-bootstrapped on first governed mutation
- **Corrupt entry**: fails closed — the governance CLI refuses to proceed until the corruption is resolved
- **Deleted journal**: regression detection is lost, but all other governance mechanisms continue to function

If the journal is lost, run `coord/scripts/gov doctor` to assess state, then manually re-establish a baseline with the next governed mutation.

#### Tamper-evidence (hash-chain)

The journal is a tamper-evident hash-chain (ENT-002, Community-tier, repo-local). Each appended event records `prev_event_hash` — the canonical hash of the immediately-preceding stored event — so any reorder, in-place tamper, or dropped event breaks a chain link and is detectable:
- **`gov doctor`** verifies chain integrity board-wide and reports a broken link as an error.
- **`gov conform`** is a read-only verb that verifies the chain end-to-end and prints the chain head plus a pass/fail verdict. This is the local self-verify; the chain head is the attestation input a future central re-hash service would re-compute. Run `coord/scripts/gov conform` (add `--json` for machine output) anytime; it never mutates governed state.
- **`gov verify-engine`** (ENT-011, Community per-team) pins the engine surface to a known-good version and detects drift from it WITHOUT signing (signing is the Enterprise half, ENT-008). `coord/scripts/gov verify-engine --pin` snapshots the current surface (the `TEMPLATE_SYNC_MANIFEST` version + its sha256 fingerprint, reusing ENT-010's conformance fingerprint, plus a per-file checksum snapshot) into `coord/engine-pin.json` — the only mutation. `coord/scripts/gov verify-engine` (no flag, add `--json`) is read-only and reports IN-SYNC or DRIFTED with the offending file paths. This is complementary to `check-template-sync`: that verifies internal manifest-vs-files consistency, while `verify-engine` verifies drift from a frozen pinned version (re-pin after an intentional engine bump).
- **Migration is non-destructive**: events predating the chain (legacy, written before ENT-002) carry no `prev_event_hash`; they validate as accepted-but-unverified "pre-chain" events rather than false tamper alarms. The first chained append on a legacy log writes an explicit, auditable `chain-anchor` event so history is never rewritten.
- **Torn-tail repair re-anchors explicitly**: a crash-truncated tail (COORD-033) is re-anchored by the auditable `journal-tail-repair` marker, so a legitimately repaired tail does not read as tamper.

### Feature Proof Verification

Feature proof entries (`path:`, `symbol:`, `text:`, `route:`) are validated at two levels:
- **Format validation** at plan-entry time (regex checks)
- **Existence verification** at landing time via `git cat-file`, `git show`, and `git grep` against the canonical branch

This means an agent can record a proof entry during development, but the entry is verified against the actual codebase before the ticket can land.

### Lock Release Guarantees

All governance mutations that acquire locks use `try/finally` blocks. If a subprocess fails (e.g., `git` or `gh` command error), the lock is still released. The governance CLI never leaves a lock held after an error.

### Doctor Repair Scope

`coord/scripts/gov doctor --fix` applies only deterministic, non-destructive repairs:
- creates missing plan stubs for `doing`/`review` tickets
- when invoked as `coord/scripts/gov doctor --fix --ticket <ticket-id>`, rebuilds mirrored board owner/status and session binding from a complete non-stale canonical doing lock for that ticket
- removes locks for tickets not in `doing` status
- reclaims stale locks (24h+ inactive)
- removes orphaned coord worktrees verified as abandoned

Outside that ticket-scoped canonical-lock rebuild path, it does not modify board ticket data, rewrite plan records, or alter governance event history.

### Limitations

The governance framework enforces the **structure** of coordination (statuses, locks, evidence, review cycles) but has inherent limitations:
- **Self-review quality**: the framework validates that review cycles exist and have required fields, but cannot verify the depth or correctness of the review content
- **Cross-repo atomicity**: if a cross-repo change lands in one repo but fails in another, there is no automated rollback; the failed portion must be handled as a new ticket
- **Orchestrator liveness**: if no agent runs `coord/scripts/gov orch`, stale locks and blocked tickets will accumulate without alert; the preflight script detects this and warns when the last orchestrator cycle is older than 24 hours
