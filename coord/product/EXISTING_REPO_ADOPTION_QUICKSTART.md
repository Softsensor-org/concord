# Existing Repo Adoption Quickstart

Status: implemented (guide) · Owner: Softsensor · Date: 2026-06-25

## Purpose

This guide shows how to apply Concord to a repository that already exists.

The adoption model is an overlay:

> Keep the repo. Keep its tests. Keep its existing delivery conventions. Add
> Concord as the governance layer for AI-assisted planning, execution, evidence,
> and closeout.

This is different from using `coord-template` as a starter template for a new
project.

## 1. Copy Or Install The Concord Layer

Use the current supported installer path for the repo.

For a local template checkout:

```bash
bash /path/to/coord-template/init.sh --repo B:app --project my-project
```

Or, if the Concord engine is already present in the repo, run:

```bash
coord/scripts/coord onboard . --dry-run
coord/scripts/coord init --dry-run
coord/scripts/coord init
```

The onboard path is read-only by default: it detects the repo shape, recommends
a governance tier and track preset, and prints next steps. `coord init` is
intended to create missing governance seams without replacing existing files;
verify the installer and target state before relying on that behavior. The
existing-repo overlay installer remains subject to the collision and rollback
limitations documented in the release backlog until its transaction tests pass.

## 2. Review The Repo Map

Open:

```text
coord/project.config.js
```

Confirm:

- repo codes match the existing repo layout;
- `integrationBranch` matches the real base branch;
- `testCommand` matches the existing local test command;
- track prefixes match the ticket naming convention;
- `coord/setup.decisions.json` captures the chosen adoption profile, phase,
  suggested tracks, suggested gates, detected repo shape, and next steps;
- no one has hardcoded local machine paths.

For a multi-repo family, update `coord/product/REPOS.md` as the human-readable
map.

## 3. Run The Readiness Doctor

Run the product-facing adoption scanner:

```bash
coord/scripts/coord onboard . --dry-run
coord/scripts/coord onboard . --dry-run --json
```

This is not the same as governance-state repair:

- `coord/scripts/coord onboard` reports adoption readiness and setup decisions.
- `coord/scripts/gov doctor` reports governance state health.

The readiness report identifies detected repo shape, package managers, tests,
requirement sources, shim state, governance setup, setup decisions generated by
`coord onboard --write`, recommended adoption profile, recommended phase, pilot
blockers, enterprise blockers, and suggested tickets.

## 4. Choose The Adoption Profile

Use the doctor recommendation as a starting point.

| Profile | Use when |
| --- | --- |
| `solo-dev` | one developer, prototype, or early repo |
| `small-team` | several developers or agents share one repo |
| `product-engineering` | product work needs requirements, tests, release evidence |
| `regulated` | URS/SRS, validation, audit, or traceability matters |
| `enterprise` | repo-family rollout or command-center view is planned |
| `production-mcp` | agents observe deployed systems through approved adapters |
| `server-bootstrap` | startup, seed, migration, replay, or backfill work is central |

Profiles set governance posture. Tracks still select the work harness.

## 5. Preserve Existing Tests And Gates

Concord should call the repo's existing tests instead of replacing them.

Example:

```js
// coord/project.config.js
module.exports = {
  repos: {
    B: {
      path: "backend",
      integrationBranch: "main",
      testCommand: "npm test"
    }
  }
};
```

If local gates are the authority, do not add GitHub Actions as a new source of
truth just because Concord is installed. If CI is already part of delivery,
document how local gates and CI relate.

## 6. File The First Governed Ticket

Use a small, low-risk ticket first.

```bash
coord/scripts/gov claim --owner <agent-handle>
coord/scripts/gov file-ticket --repo X --type docs --pri P1 \
  --description "Document the existing repo adoption baseline and first gate."
```

Then add a ticket prompt under:

```text
coord/prompts/tickets/<TICKET-ID>.md
```

Run:

```bash
node coord/board/board.js sync
node coord/board/board.js validate
```

## 7. Execute One Ticket End To End

Start:

```bash
coord/scripts/gov start <TICKET-ID> --owner <agent-handle>
```

Make the smallest useful change and record evidence:

```bash
coord/scripts/gov add-feature-proof <TICKET-ID> --proof-path <path>
coord/scripts/gov add-repo-gate <TICKET-ID> \
  --command "<existing local gate command>" \
  --result pass \
  --note "Existing repo gate passed."
```

Add review cycles and requirement closure, then close locally when there is no
PR:

```bash
coord/scripts/gov submit <TICKET-ID> --pr "local-review (no PR)"
coord/scripts/gov finalize <TICKET-ID> --no-pr \
  --landed "Local reviewed change; existing gate passed." \
  --source-commit no-pr-local
```

## 8. Treat Doctor Findings As Work, Not Noise

Doctor findings should become governed tickets when they matter.

Examples:

- missing shims -> shim drift validator or setup ticket;
- missing requirements -> requirements baseline ticket;
- missing test command -> project config ticket;
- GitHub Actions detected but local gates are authoritative -> docs cleanup;
- enterprise blockers -> enterprise hardening roadmap ticket.

Do not silently fix everything in setup. Setup prepares. Governed tickets decide
and deliver.

## 9. Run Business Discovery Before High-Risk Refactors

For existing repos with unclear business context, run a business-discovery pass
before advising major refactors, domain rewrites, schema changes, or generalized
platform extraction.

The discovery protocol records source-backed facts, business objects, field
rules, workflows, configuration surfaces, contradictions, hypotheses, questions,
and promotion candidates. Code is treated as evidence, not automatically as
intent. Implementation-only findings stay `observed` or `candidate` until an
authoritative source or business owner promotes them.

Use the protocol in [`BUSINESS_DISCOVERY_PROTOCOL.md`](BUSINESS_DISCOVERY_PROTOCOL.md)
and the schema in [`business-discovery.schema.json`](business-discovery.schema.json).

## 10. Pilot Boundary

Pilot-ready means the repo has Concord installed, a profile is selected or
recommended, at least one ticket has completed with evidence, local gates are
clear, and limitations are explicit.

Pilot-ready does not mean SSO, RBAC, KMS/HSM signing, tenant isolation, or
enterprise procurement readiness are implemented. Track those separately.

## 11. Adoption Checklist

- [ ] `coord/project.config.js` reviewed.
- [ ] `coord/setup.decisions.json` reviewed.
- [ ] `coord/scripts/coord doctor --dir .` run.
- [ ] Adoption profile chosen or accepted from doctor.
- [ ] Existing test commands preserved.
- [ ] Agent shims point to `coord/GOVERNANCE.md`.
- [ ] First ticket prompt added.
- [ ] First governed ticket completed.
- [ ] Business discovery run planned for high-risk existing-repo changes.
- [ ] Pilot blockers and enterprise blockers separated.
